# 需求文档（need.md）

## 1. 项目目标

构建一个面向 PostgreSQL 内核学习与观测的可视化平台。当前阶段以 **Web MVP** 为主，但运行架构必须明确为：

```text
前端浏览器 -> 后端服务 -> PostgreSQL / 数据库宿主机 / 采集器
```

平台的核心不是“浏览器直连数据库”，而是“后端统一连接、统一编排、统一观测”。

## 2. 架构约束

### 2.1 连接约束

- 浏览器不直接连接 PostgreSQL。
- 浏览器不直接连接数据库所在机器。
- 所有数据库连接由后端发起并维护。
- 所有宿主机扫描、实例发现、集群拉起都由后端执行。

### 2.2 状态约束

- 工作区中的项目、集群、节点、宿主机、任务状态都由后端持久化。
- 前端只保存临时 UI 状态，不作为真实连接状态来源。
- 观测页展示的数据来源必须标注为“实时 / 快照 / 暂无数据 / 接口异常”。

## 3. 核心用户场景

1. 教学演示：点击按钮一键拉起单机或复制集群，直接进入观测。
2. 联调排障：手动接入某个数据库节点，由后端连接后查看 SQL / WAL / CLOG / 锁 / 事务状态。
3. 主机托管：添加一台运行 PostgreSQL 的机器，由后端自动扫描实例并导入观测。
4. 架构理解：在“项目 -> 集群 -> 节点”之间快速切换，理解复制链路与节点内部状态。

## 4. 功能需求

### 4.1 工作区层级

- 支持“项目”对象：创建、删除、选择。
- 支持“集群”对象：创建、删除、选择，归属项目。
- 支持“节点”对象：新增、删除、编辑连接参数，归属集群。
- 支持“组件”对象：创建、删除、与集群建立关联。
- 支持“宿主机”对象：新增、删除、查看扫描状态，归属项目。

### 4.2 数据库接入方式

1. **手动添加数据库节点**
- 用户录入主机、端口、用户、密码、数据库名。
- 后端验证并建立连接。
- 成功后进入集群主页与节点观测。

2. **一键拉起单机 PostgreSQL**
- 后端创建单实例运行环境。
- 自动生成工作区节点与默认观测配置。
- 自动连接并进入观测。

3. **一键拉起主备复制集群**
- 后端创建 primary / standby。
- 自动完成复制参数初始化与复制关系检查。
- 自动生成拓扑与同步状态。

4. **一键拉起逻辑复制集群**
- 后端创建 publisher / subscriber。
- 自动创建 publication / subscription。
- 自动校验逻辑复制状态并进入观测。

5. **添加数据库宿主机并自动发现实例**
- 用户录入机器地址与访问方式（优先 SSH）。
- 后端扫描 PostgreSQL 实例、版本、端口、数据目录、服务信息。
- 用户可选择导入某个实例，或按策略自动导入。

### 4.3 集群可观测

- 集群主页展示：
  - 集群总览指标（节点总数、在线数、异常数）
  - 复制拓扑图（可交互）
  - 同步状态看板（来自后端 `cluster overview`）
- 复制链路可视化支持边与节点点击查看详情。
- 自动拉起成功后默认展示集群健康与拓扑状态。

### 4.4 节点可观测

点击节点后进入节点主页，至少支持：
- SQL 控制台
- WAL 查看
- CLOG 查看
- 锁等待图
- 事务状态
- 内存结构

### 4.5 任务中心

- 所有“一键拉起”“宿主机扫描”“实例导入”都以任务形式执行。
- 任务需展示 `pending / running / success / failed` 状态。
- 任务失败时必须给出错误原因与建议。
- 任务成功后应自动刷新工作区并跳转到可观测页面。

## 5. 非功能需求

- 可用性：页面逻辑清晰，中文界面优先。
- 安全性：数据库与 SSH 凭据仅在后端使用与持久化。
- 可扩展性：拉起器支持后续扩展 `docker / local / k8s` provider。
- 一致性：`need / design / apply / README / plan` 文档口径一致。

## 6. 当前范围判断

### 6.1 当前已具备

- 工作区基本模型
- 手动数据库节点接入
- 后端驱动的集群 / 节点观测

### 6.2 当前待补齐

- 真实可用的一键拉起能力
- 宿主机资源模型与自动发现能力
- 任务编排、回滚与自动化回归

## 7. 本阶段验收标准

1. 浏览器不再承担任何直连数据库职责，所有连接均走后端。
2. 手动添加数据库节点可稳定进入观测。
3. 单机、主备、逻辑复制三类“一键拉起”至少各完成一个可用闭环。
4. 添加宿主机后可扫描并导入至少一个 PostgreSQL 实例。
5. 所有关键流程都有任务状态、错误提示和工作区刷新。
6. 文档与实际实现一致，不夸大已完成范围。
